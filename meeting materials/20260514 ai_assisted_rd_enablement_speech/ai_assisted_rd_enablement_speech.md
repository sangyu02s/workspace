# 一、简要摘要：本次分享讲什么？

这次分享围绕 **“AI 如何辅助研发工作”** 展开，不追求把所有前沿概念一次讲完，而是希望帮助大家建立一套可以落地的认知框架：

> **AI 不是替代开发者的人，而是放大开发者能力的工具。真正会用 AI 的开发者，不是把判断交出去，而是更会定义问题、组织上下文、拆解任务、验证结果。**

整场内容分为五个部分。

---

## 第一部分：AI 发展与代码辅助开发演进

这一部分先从过去三年 AI 能力的变化讲起：  
AI 从一个“会聊天的工具”，逐步发展成能够理解上下文、调用工具、参与流程、甚至执行任务的协作者。

然后聚焦代码开发领域，梳理 AI 编程工具从 **自动补全、代码解释、多工具协作、代码库理解** 到 **Agent 化开发** 的演进过程。

这一部分的核心结论是：

> AI 编程工具的工作单位，正在从“一行代码 / 一个函数”，变成“一个变更集 / 一个任务 / 一个问题单”。

---

## 第二部分：AI Agent：概念、机制与实践

这一部分解释什么是 AI Agent，以及它和普通聊天助手有什么区别。

普通 AI 助手更像“你问一句，它答一句”；  
AI Agent 则更像“你给它一个目标，它会尝试拆任务、调用工具、执行动作、观察结果，并不断修正”。

这一部分会结合 **opencode CLI** 来说明：

- `/init` 为什么重要
- `AGENT.md / AGENTS.md` 的作用是什么
- Prompt、Context、LLM、MCP、Skills、Subagent、Human-in-the-loop 等概念分别在开发流程中扮演什么角色
- 一次真实开发请求是如何在 Agent 内部流动的
- ReAct 循环如何让 Agent “边推理、边调用工具、边观察结果”

这一部分的核心结论是：

> AI Agent 不是一个单点能力，而是一条由 Prompt、Context、LLM、Skills、MCP 工具调用和人工验收串起来的任务执行链路。

---

## 第三部分：OpenSpec：规格驱动需求开发

这一部分回答一个很关键的问题：

> 为什么不能直接让 Agent 做需求开发？

因为真实研发中最难的往往不是“写代码”，而是：

- 需求是否清楚
- 边界是否明确
- 隐性业务规则是否说清楚
- 验收标准是否可验证
- 高风险决策是否有人拍板

OpenSpec 的价值，就是把一句模糊的 Prompt，变成一组可讨论、可审查、可执行、可验证、可追溯的规格文档。

这一部分会介绍 OpenSpec 的核心文档结构：

- `proposal.md`
- `specs/**/*.md`
- `design.md`
- `tasks.md`

也会用“用户列表接口增加分页能力”这个例子，演示从 **探索 → New → Continue → Apply → Verify → Archive** 的完整需求开发流程。

这一部分的核心结论是：

> OpenSpec 不是让我们多写文档，而是让 AI Agent 在写代码前先拥有清晰目标、明确边界和可验证任务。

---

## 第四部分：赋能总结：价值、误区与拥抱方式

这一部分总结 AI 能给研发带来什么，以及我们对 AI 常见的误区。

AI 能帮助我们：

- 降低重复劳动
- 提升理解效率
- 前置质量检查
- 加速需求闭环

但 AI 也有明显边界。它不适合独立承担：

- 架构方案拍板
- 高风险业务逻辑修改
- 没有上下文的需求开发
- 没有测试的大规模重构
- 代替人工验收和上线责任

这一部分也会用一个互联网笑话提醒大家：

> **AI 有时候像一个刚入职三天，但语气像 CTO 的实习生。**  
> 它最大的问题不是会错，而是错的时候也很像对的。

这一部分的核心结论是：

> 让 AI 多干活，但别让 AI 独自拍板；让 AI 提效率，但别让 AI 替你负责。

---

## 第五部分：AI Agent 协作开发范式

最后一部分给出一套更完整的 AI 结对编程协作范式。

我们把人与 AI 的协作拆成：

```text
计划 → 实施 → 验证 → 学习
```

人在其中始终承担决策与把关职责，AI 负责需求复述、方案生成、增量实施和自测。

这一部分的核心结论是：

> 人负责决策与验证，AI 负责方案、实施与自测。局部问题走小循环修复，方向错误再回到方案阶段重来。

# 三、第一部分：AI 发展与代码辅助开发演进

## 1. 过去三年：AI 从“会对话”走向“会执行”

过去三年，AI 最大的变化，不只是模型变强了，也不是回答越来越像人了。

更重要的是，它完成了四次连续跃迁。

第一阶段叫 **对话式入口化**。  
ChatGPT 出现之后，AI 第一次变成了一个普通人可以直接使用的聊天窗口。以前 AI 更多藏在搜索、推荐、广告、风控系统背后，我们很少直接感知它。而 ChatGPT 让大家第一次感觉到：

> 原来我可以直接用自然语言和机器协作。

比如以前你想让机器帮忙总结文章，可能要找工具、写脚本、复制粘贴。  
现在可以直接说：

```text
帮我总结这篇文章。
帮我解释这段代码。
帮我写一封邮件。
帮我把这个需求拆成开发任务。
```

这就是“对话式入口化”的价值：降低门槛。

第二阶段是 **多模态与长上下文化**。  
AI 不再只能看一小段文本，而是可以看图片、截图、代码、日志、长文档、表格、PDF，甚至一部分代码库上下文。

这对研发非常关键。因为真实的研发问题通常不是一句话能说清的。

比如线上报错，真正需要看的可能是：

```text
错误日志
相关代码
接口文档
最近改动
测试结果
项目规范
```

AI 能接收更多上下文后，它就更接近真实工作场景。

第三阶段是 **工具调用与工作流化**。  
AI 不再只是回答问题，而是能调用工具。

比如：

```text
读文件
跑命令
查数据库
调用 API
访问 GitHub
运行测试
```

以前 AI 可能告诉你：

```text
你可以运行 mvn test 看看测试是否通过。
```

现在 Agent 可以直接跑：

```bash
mvn test
```

然后根据结果继续分析。

第四阶段就是 **Agent 化**。  
Agent 化的关键，不是它写的代码更多，而是它能围绕一个目标持续推进。

比如你不再只是问：

```text
帮我写一个分页函数。
```

而是说：

```text
请阅读这个项目，找到用户列表接口，增加分页参数，兼容旧调用，补充单元测试，运行测试，并说明影响范围。
```

这时 AI 的角色就从“回答问题”，逐步走向“协作完成任务”。

所以这一页最重要的一句话是：

> **AI 的价值正在从“生成内容”，转向“完成任务”。**

## 2. 代码辅助开发：从“自动补全”到“任务托管”

代码开发是生成式 AI 影响最早、最深的领域之一。

我们可以把过去几年的 AI 编程工具，大致分成四个阶段。

第一个阶段是 **补全时代**。

这个阶段的代表是 GitHub Copilot 这一类工具。它最常见的能力是：

```text
行级补全
函数级补全
样板代码生成
API 调用建议
局部逻辑补全
```

比如你刚写下：

```java
public User getUserById(
```

AI 可能已经猜到后面要做：

```text
接收 userId
查数据库
处理空值
抛异常
返回 UserDTO
```

这时候 AI 更像一个很聪明的 IDE 提示。  
以前 IDE 提示变量名，现在 AI 能补完一段逻辑。

第二个阶段是 **代码解释 + 多工具时代**。

这时 AI 不只是帮你写代码，还能帮你读代码、解释报错、生成测试、辅助重构。

比如你接手一段祖传代码，里面一堆 if-else，可以问：

```text
这段代码主要在做什么？
有哪些潜在 bug？
能不能帮我拆成几个更清晰的函数？
是否可以补一些单测？
```

它不一定完全正确，但它能帮你快速建立第一版理解。

第三个阶段是 **代码库级理解**。

这一步非常重要。AI 不再只看当前文件，而是开始理解：

```text
仓库结构
模块依赖
调用链
Issue
日志
错误栈
项目约定
```

比如线上报错：

```text
NullPointerException in OrderService.createOrder
```

以前我们要自己翻日志、查调用链、看最近改动。  
现在可以让 AI 结合代码库分析：

```text
这个异常可能从哪里传进来的？
相关文件有哪些？
最近哪些改动可能影响这里？
修复时要注意哪些调用方？
```

第四个阶段是 **Agent 化开发**。

这个阶段的特点是，AI 的工作单位从“一段代码”变成“一个任务”。

比如你可以说：

```text
给这个接口加分页参数，兼容原有调用，补单测，跑测试，最后整理变更说明。
```

Agent 会尝试：

```text
读代码 → 找入口 → 改实现 → 补测试 → 跑测试 → 修失败 → 输出变更说明
```

这个阶段，AI 就像一个可以托管小任务的实习工程师。

但还是要强调：  
它能执行，不代表它能负责。

---

## 3. 四个阶段背后的本质变化

这几个阶段背后，其实是开发者和工具关系的变化。

补全时代：

```text
AI 提示，我来判断。
```

解释和多工具时代：

```text
AI 分析，我来决策。
```

代码库级理解时代：

```text
AI 理解项目，我来审查。
```

Agent 化开发时代：

```text
AI 执行任务，我来验收。
```

所以，开发者的角色也在变化。

我们不只是逐行写代码的人，而越来越像：

```text
问题定义者
方案判断者
上下文组织者
质量把关者
最终责任人
```

这里可以用一句话总结：

> **编程活动的单位，正在从 token / 函数，变成变更集 / 任务 / 问题单。**

# 四、第二部分：AI Agent：概念、机制与实践

## 4. 什么是 AI Agent？

普通 AI 助手和 AI Agent 最大的区别，可以用一句话概括：

> 普通 AI 助手是“你问一句，它答一句”；AI Agent 是“你给它一个目标，它尝试完成一串任务”。

比如普通 AI 助手，你问：

```text
帮我写一个分页查询函数。
```

它通常返回一段代码，你自己复制、修改、运行、测试。

而 AI Agent，你可以说：

```text
给用户列表接口加分页能力，兼容旧调用，补充单测，跑测试，并整理变更说明。
```

它会尝试：

```text
读取相关代码和文档
分析接口和调用方
修改实现代码
补充测试用例
运行测试并修复失败
输出结果或提交 PR 等待 Review
```

一个通俗类比是：

> 普通 AI 助手像“答疑同学”，Agent 更像“可以接小任务的实习生”。

但是这句话后半句更重要：

> 它能执行，不代表它能负责。

最终质量、风险和决策，仍然需要开发者把关。

## 5. opencode CLI：让 Agent 进入项目上下文

接下来我们用 opencode CLI 做例子。

它不是多一个网页聊天窗口，而是让 Agent 进入你的本地项目目录，理解代码结构和工程约定。

安装可以用：

```bash
npm install -g opencode-ai
opencode --version
```

进入项目目录后运行：

```bash
cd /path/to/java-service
opencode
```

进入后，第一步建议执行：

```text
/init
```

为什么要先 `/init`？

因为 Agent 不是只靠聪明工作，它还需要项目上下文。

`/init` 会帮助 Agent 生成或更新类似 `AGENT.md / AGENTS.md` 的文件。  
这个文件可以理解成：

> 写给 AI Agent 的项目说明书。

它会告诉 Agent：

```text
项目技术栈是什么
目录结构怎么分层
如何运行测试
哪些文件不能动
新增接口要遵守什么规范
高风险变更要不要先问人
```

比如 Java 微服务项目，可以写：

```markdown
## Tech Stack
Java 17 / Spring Boot 3.x / Maven

## Common Commands
mvn test
mvn -Dtest=UserServiceTest test

## Coding Rules
- Controller 不写复杂业务逻辑
- Service 负责事务边界
- 新增接口必须补 DTO 和参数校验
- 修改公共方法要检查调用方

## Safety Rules
- 不改 application-prod.yml
- 不碰密钥、权限、支付、数据删除逻辑
- 高风险变更先出方案，等待人工确认
```

这样 Agent 进入任务时，就不是一个通用助手，而是一个知道项目规则的协作者。

日常使用时，也建议先让它分析，而不是直接改：

```text
帮我分析 UserController 中用户列表接口的调用链，
说明会涉及哪些 Service / DTO / Repository，
先不要修改代码。
```

更稳的节奏是：

```text
分析现状 → 输出计划 → 确认风险 → 再让 Agent 修改代码与补测试。
```

## 7. 模型与输入：AI Agent 到底看见了什么？

理解 Agent 之前，先理解模型到底“看见”了什么。

第一个概念是 **LLM**。  
LLM 是 Agent 背后的大脑，负责理解语言和代码，推理任务意图，生成回答或下一步动作。

它像一个能力很强的临时队友，但只能基于当前看到的信息工作。

第二个概念是 **Prompt**。  
Prompt 是你交给模型的任务说明。

差一点的 Prompt 是：

```text
帮我看看这段代码。
```

更好的 Prompt 是：

```text
请分析 UserService.createUser 的逻辑，
重点说明入参校验、异常分支和潜在风险。
先不要改代码，只输出分析结论。
```

好的 Prompt 等于：

```text
目标 + 约束 + 输出格式
```

第三个概念是 **System Prompt**。  
它是更高优先级的行为规则，规定模型的角色、边界和输出方式。

比如：

```text
你是一个谨慎的 Java 后端开发助手。
涉及生产配置、密钥、权限、支付、数据删除等高风险变更时，
不要直接修改代码，必须先输出方案并等待人工确认。
```

如果用户说“直接改生产配置”，但系统规则说不能直接改，那 Agent 应该优先遵守系统规则。

第四个概念是 **Context**。  
Context 是模型当前能看到的资料包，包括：

```text
当前对话
项目文件
代码片段
错误日志
AGENT.md
README
接口文档
测试结果
历史修改记录
```

同一句话：

```text
帮我给用户列表加分页。
```

没有 Context 时，模型只能生成通用示例。  
有项目 Context 时，模型可以结合 Controller、Service、DTO、Test 和项目规范分析。

第五个概念是 **Context Window**。  
它是模型一次能看到的“桌面大小”。

桌子越大，能放的资料越多。  
但重点不是塞满桌子，而是把真正相关的资料放上去。

第六个概念是 **Token**。  
代码、日志、文档都会被切成 Token。Token 会占用上下文窗口，也会影响成本和响应速度。

所以对 Coding Agent 来说，真正重要的不是只换一个更强模型，而是：

```text
任务说清楚
上下文给准确
规则边界讲明白
```

## 8. AI Agent 扩展机制：MCP、Skill、Subagent

这一页重点讲三个东西：

```text
MCP 解决“接什么工具”
Skill 解决“怎么稳定做事”
Subagent 解决“谁专门负责哪类任务”
```

### MCP

MCP 像给 Agent 装上的通用插座。  
它让 Agent 能通过统一方式连接外部工具和数据源。

比如：

```text
GitHub
数据库
知识库
内部系统
文件系统
```

### Skill

Skill 是可复用的做事方法包。

如果说模型会思考，那 Skill 就是在告诉它：

```text
遇到这类任务，最好按什么套路做。
```

比如：

```text
PR Review Skill
测试生成 Skill
周报生成 Skill
接口文档生成 Skill
```

它像团队沉淀下来的一套 SOP。

### SKILL.md

`SKILL.md` 是 Skill 的说明书，写清楚：

```text
步骤
规则
模板
示例
输出格式
```

例如一个 PR Review Skill 可以规定：

```text
先看 diff
再查风险点
最后按模板输出 review 结论
```

### Subagent

Subagent 是被主 Agent 调度的专职小助手。

比如主 Agent 可以把不同子任务分出去：

```text
代码库搜索员：专门找相关文件
测试专家：专门补测试
代码审查员：专门挑风险
文档整理员：专门写说明
```

Skill 和 Subagent 的区别可以这样理解：

```text
Skill 是方法包，沉淀流程；
Subagent 是专职角色，负责分工。
```

---

## 9. 一次真实请求，AI Agent 是如何工作的？

我们用一个具体需求串起来：

```text
请帮我给用户列表接口增加分页能力，兼容旧调用，补充单元测试，并说明可能影响哪些文件。
```

这条链路只看几个核心概念：

```text
Prompt → Context → LLM → Skills → MCP → Human-in-the-loop
```

### Prompt

Prompt 给目标。  
它告诉 Agent：

```text
这次要完成什么
不能破坏什么
最后要交付什么
```

### Context

Context 给材料。  
它告诉 Agent：

```text
项目当前是什么样
团队规则是什么
历史约定是什么
相关代码在哪里
```

### LLM

LLM 负责推理。  
它理解需求，判断下一步应该做什么。

### Skills

Skills 给套路。  
比如按照团队 SOP：

```text
先分析影响范围
再出计划
再补测试
再写变更说明
```

### MCP

MCP 连接工具。  
让 Agent 能真正读文件、查仓库、跑命令、访问外部系统。

### Human-in-the-loop

最后由人把关。  
人来 Review diff，判断风险，决定是否接受。

---

## ReAct 核心循环

这一页还补充了一个重要概念：**ReAct**。

ReAct 可以理解成：

```text
Reason → Act → Observe
```

也就是：

```text
想一想 → 做一下 → 看结果
```

不是一次性猜答案，而是循环：

```text
先推理
再调用工具
再观察结果
再继续推理
```

例如：

```text
先找 UserController
再读 UserService
再跑 mvn test
失败后继续修正
```

这就是 Agent 之所以能逐步推进任务的原因。

# 五、第三部分：OpenSpec：规格驱动需求开发

## 10. 为什么不能直接让 Agent 做需求开发？

直接让 Agent “开写”很诱人，但风险很大。

真正痛点不在“AI 会不会写代码”，而在：

```text
需求有没有对齐
边界有没有说清
结果能不能验证
```

第一个问题是 **需求太模糊**。  
比如：

```text
帮我优化用户列表接口。
```

到底是加分页、加筛选、提升性能，还是调整返回字段？  
如果没有说明，Agent 只能猜。

第二个问题是 **范围边界不清**。  
你只想给列表接口加分页，Agent 可能顺手改 DTO、导出接口、前端字段、接口文档，甚至通用分页组件。

第三个问题是 **隐性业务规则没写出来**。  
比如用户列表默认只能看本部门数据。  
如果 Agent 不知道这条规则，就可能把跨部门数据也查出来。

第四个问题是 **验收标准缺失**。  
Agent 可能代码能跑就说完成了，但团队真正关心的是：

```text
测试补齐了吗？
兼容老调用了吗？
性能有没有退化？
接口文档更新了吗？
```

第五个问题是 **高风险决策不适合 Agent 自己拍板**。  
比如：

```text
数据库结构
权限模型
事务边界
支付逻辑
生产配置
数据删除
```

这些需要人类负责。

所以这一页的结论是：

> **直接 Prompt 不等于完整需求。真正需要补上的，是范围、规则、验收标准、风险边界。**

这正是 OpenSpec 的价值。

---

## 11. OpenSpec 是什么？

OpenSpec 是面向 AI Coding Assistant 的轻量级规格驱动开发框架。

它的核心思想是：

```text
先 Agree，再 Build。
```

也就是先对齐要构建什么，再让 Agent 去实现。

直接对 Agent 说“帮我实现这个需求”，容易出现：

```text
需求只在聊天里，换会话就丢
Agent 自行脑补边界和验收标准
Review 只能看代码，看不到意图
需求、设计、任务、实现难以追溯
```

OpenSpec 把需求沉淀到仓库里，让它成为团队共享的事实源。

目录可能是：

```text
openspec/
├─ specs/
├─ changes/
│  └─ add-pagination/
└─ changes/archive/
```

OpenSpec 的核心文档包括四个：

### proposal.md

回答：

```text
为什么做？
要做什么？
影响范围是什么？
```

### specs/**/*.md

描述系统行为变化和验收场景。  
比如：

```text
未传分页参数时保持旧行为
传入 page=1,pageSize=20 时返回第一页数据并包含 total
```

### design.md

说明技术方案、风险和权衡。  
复杂需求尤其需要它。

### tasks.md

把需求拆成 Agent 可执行清单。  
比如：

```text
确认接口入口
增加 DTO 参数
修改 Service
补测试
更新文档
```

---

## 12. 用 OpenSpec 驱动一次需求开发：上半场

示例需求：

```text
给用户列表接口增加分页能力，兼容旧调用，补充单元测试，并说明可能影响哪些文件。
```

### Explore：先把问题问清楚

命令：

```text
/opsx:explore 用户列表接口需要增加分页能力；先分析现状并列出待确认问题
```

这一阶段不创建正式变更文件，也不修改代码。  
主要做：

```text
澄清需求
识别歧义
发现风险点
理解现状
```

会生成一份问题清单：

```text
分页参数用 page/pageSize 还是 offset/limit？
page 从 0 还是 1 开始？
不传分页参数是否保持旧行为？
是否需要返回 total？
是否影响导出接口和前端调用？
```

Explore 像需求评审前的白板讨论。  
先把坑挖出来，再决定如何建 spec。

### New：创建正式变更空间

命令：

```text
/opsx:new add-user-list-pagination
```

生成：

```text
openspec/changes/add-user-list-pagination/
├── proposal.md
├── design.md
├── tasks.md
└── specs/
```

这一步像立项建文件夹。  
从这时开始，需求不再只是几句 Prompt，而是仓库中一个有编号、有文档、有上下文的正式变更。

### Continue：继续补全规格

命令：

```text
/opsx:continue add-user-list-pagination
```

补充：

```text
默认值
兼容策略
验收场景
任务拆解
影响范围
```

比如把“支持分页”具体化成：

```text
page 默认 1
pageSize 默认 20
pageSize 最大 100
返回 total
补正常 / 边界 / 兼容性测试
```

---

## 13. 用 OpenSpec 驱动一次需求开发：下半场

### Apply：按 tasks.md 实现

命令：

```text
/opsx:apply add-user-list-pagination
```

Agent 按任务清单改代码，而不是自由发挥。

可能修改：

```text
UserController.java
UserQueryDTO.java
UserService.java
UserServiceTest.java
docs/api/user-list.md
```

这一步像照着施工图施工。

### Verify：确认代码、测试和规格一致

命令：

```text
/opsx:verify add-user-list-pagination
```

或者：

```bash
openspec validate --strict
mvn test
```

验证不是只看代码能不能跑，而是看：

```text
是否实现了 spec？
测试是否覆盖了 scenario？
改动是否超出范围？
是否还有兼容风险？
```

### Archive：沉淀为系统事实

命令：

```text
/opsx:archive add-user-list-pagination
```

完成后，临时变更进入 archive，最终规格合并到长期 specs。

以后再有人问：

```text
用户列表分页为什么这么设计？
```

不用翻聊天记录，直接看 archive 和 specs。

# 六、第四部分：赋能总结

## 14. 如何看待 AI 辅助研发

这一页想表达一个观点：

> AI 是能力放大器，不是自动驾驶。

AI 的价值不是替我们少思考，而是把重复劳动交给机器，把问题定义、方案判断、质量把关留给人。

AI 能带来：

```text
降低重复劳动
提升理解效率
前置质量检查
加速需求闭环
```

但常见误区包括：

```text
AI 会写代码，所以能替代开发者
AI 说得很自信，所以它一定是对的
上下文越多越好
有了 Agent 就可以跳过 Review
```

这里可以讲一个互联网笑话：

> AI 有时候像一个刚入职三天，但语气像 CTO 的实习生。

还有一句更经典：

> AI 最大的问题不是它会错，而是它错的时候也很像对的。

所以我们应该：

```text
把 AI 当研发搭子，而不是外包团队
先让 AI 分析和计划，再让它执行
把需求说清楚，把上下文给准确
用 AGENT.md、OpenSpec、tasks.md、测试、Lint、Review 和 Guardrails 约束它
```

## 15. 软件开发视角：AI 擅长什么，不适合什么？

AI 在软件开发中很有用，但要用在合适的地方。

### AI 擅长什么？

适合有上下文、有规则、可验证的任务，比如：

```text
读懂陌生代码
生成低风险样板代码
补测试和边界用例
解释报错和日志
整理变更说明
执行有 SOP 的任务
```

例如：

```text
帮我梳理 UserService 的调用链。
根据这个方法补几个单元测试。
根据这次 diff 写一份 PR 描述。
帮我分析这段错误日志可能来自哪里。
```

这些任务共同特点是：  
目标相对清楚，输入材料明确，结果可以验证。

### AI 不适合什么？

AI 不适合独立承担缺上下文、强决策、高风险、难验证的事情。

比如：

```text
独立拍板架构方案
直接修改权限、支付、生产配置、数据删除逻辑
猜业务隐性规则
无测试地大规模重构
代替人工验收
处理模糊需求直接开写
```

这些事情如果交给 AI 独立完成，风险会很高。

所以使用原则是：

> 让 AI 做资料整理、代码生成、测试补充、自检说明；让人负责需求判断、方案取舍、风险验收、最终上线责任。

# 七、第五部分：AI Agent 协作开发范式

## 16. AI 结对编程协议：计划 - 实施 - 验证 - 学习

最后，我们给出一套更完整的人机协作范式。

这套范式强调：

```text
人始终承担决策与验收职责
AI 负责需求复述、方案生成、增量实施与自测
```

第一阶段是 **需求理解与任务拆分**。  
人提供业务背景、目标、约束和优先级；AI 复述需求、识别歧义、拆子任务；人确认拆分结构和 Done criteria。

第二阶段是 **方案设计与风险评估**。  
AI 给出可选方案和风险点，人判断方案是否可行。这里要设置硬 Gate：

> 人明确说“按此方案执行”，AI 才能进入下一步。

第三阶段是 **增量实施与验证**。  
AI 按确认后的方案增量修改代码并自测。人进行快速 review，检查逻辑、安全和设计一致性。

第四阶段是 **学习与沉淀**。  
对任务做复盘，把有效 Prompt、关键上下文、风险点沉淀到 AGENT.md、SKILL.md、OpenSpec 或团队知识库。

---

## 17. AI Agent 协作开发范式：简洁版流程图

这一页是最简洁的流程。

主线是：

```text
人给背景与目标
→ AI 复述并拆任务
→ AI 生成方案
→ 人确认方案
→ AI 增量实施 + 自测
→ 人验证
→ 完成 / 沉淀
```

这里有两个反馈回路。

第一个是方案阶段：

```text
如果人不确认方案，回到 AI 生成方案。
```

第二个是验证阶段：

```text
如果只是局部问题，回到 AI 增量实施 + 自测。
如果方向错了，回到 AI 生成方案。
```

这套范式最重要的不是让 AI 一次做完，而是让人与 AI 形成稳定节奏：

```text
先对齐
再决策
再增量实施
再验证
必要时小步回退
```
